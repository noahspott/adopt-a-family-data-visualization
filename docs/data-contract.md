# BuildWithChrist.ai Challenge Brief: Shared Data Contract

This document is the single source of truth for every data type used across all tracks.

## TypeScript Interfaces

### `FamilyStatus`
```ts
export type FamilyStatus =
| "unmatched"
| "matched"
| "packet-sent"
| "closeout"
| "fulfilled"
| "exception";
```
| Value | Meaning |
| :--- | :--- |
| `unmatched` | Family intake complete; waiting for a sponsor. |
| `matched` | Sponsor assigned but gifts not yet purchased. |
| `packet-sent` | Gift packet assembled and shipped / dropped off. |
| `closeout` | Drop-off received; receipt/closeout in progress. |
| `fulfilled` | Family confirmed receipt — done!. |
| `exception` | Needs coordinator attention (duplicate, withdrawal, etc.). |

### `FamilyMember`

## Status State Machine
```text
unmatched -> matched -> packet-sent -> closeout -> fulfilled
                |            |              |
                |            v              |
                +-------> exception <-------+
                             |
                             v
                         (unmatch) -> unmatched
```
The `closeout` state represents: items dropped off, coordinator verifying counts/labels, receipt sent to sponsor, final confirmation before marking `fulfilled`.

## Labeling Rule
Label format: Church Abbreviation + Family Number + Family Last Name + Box Count
*   **Example:** `MH-12 Smith — Box 1 of 2`
*   **Bikes:** Bikes are labeled using the same Church Abbrev + Family # convention with a "BIKE" suffix.

---

## Extensibility
All three primary tables (`families`, `sponsors`, `packet_checklists`) include a `metadata` jsonb column. This column is reserved for future features such as:
*   AI-powered personalization and gift recommendations
*   Language preferences and accessibility needs
*   Expanded demographic data for reporting
*   Voice-note transcription summaries
*   Affinity matching scores

*Teams may read/write `metadata` freely. The schema of its contents is not enforced — treat it as a flexible key-value store.*
